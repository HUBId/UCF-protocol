# Breaking Changes Policy

## Breaking changes require a MAJOR version bump
- Removing fields from existing messages.
- Changing the meaning or semantics of existing fields.
- Altering required field semantics or cardinality.
- Renaming enums or existing enum values.

## Non-breaking changes qualify for MINOR version bumps
- Adding new optional fields.
- Adding new enum values that preserve existing semantics.
- Adding new messages or new .proto files.
